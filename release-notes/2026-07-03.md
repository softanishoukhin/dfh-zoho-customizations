First day using this repo — all tasks below were completed and deployed *before* this repo

## Deployed to production today

1. **Amber Integration — Round 2** (`crm/functions/trips/ZP-TBD-1_amber_integration_fix/`)
   Fixed the trigger-timing gap where a trip created and completed with no intermediate edit
   never got an Amber job created (and therefore never got completion sent). Verified against
   all 5 intake types (First Call, Hospital, Police, Wholesale Local, Wholesale Airport) with
   tagged test records, since removed.

2. **Police Intake fixes** (`creator/functions/nok_intake/ZP-TBD-2_police_intake_fixes/`)
   - Deceased Name was wrongly required for Police at both the widget and form level —
     fixed in both places, matching Hospital's "unknown identity at intake" allowance.
   - The Trip's Police Station field was reading from the wrong form field (the free-text
     officer-name field instead of the station dropdown) — fixed.

3. **Police/Hospital Sales Order fallback** (`crm/functions/deals/ZP-TBD-3_police_hospital_sales_order_fallback/`)
   Both Sales Order creation functions looked up their Trip via the old single-Deal-per-Trip
   model and silently produced nothing for multi-deceased Deals. Added a Deceased_Pickups-based
   fallback, gated on the Trip actually being Completed.

4. **Police Account lookup** (`crm/functions/deals/ZP-TBD-4_police_account_lookup/`)
   `onDeceasedPickupCreate`'s Police branch now resolves the station name to a real Account
   (`Deals.Police_Account`), mirroring how the Hospital branch already resolves
   `Selected_Hospital`.

5. **Trip completion + Driver App UI** (`creator/functions/driver_app/ZP-TBD-5_trip_completion_and_ui_fixes/`)
   `completeJob` now advances each multi-deceased child Deal's Stage to "Trip Completed" for
   both Police and Hospital (previously Hospital-only, and even then only set
   Is_Deceased_In_Our_Care, never advanced Stage) — this is what makes item 3 actually reachable
   in production instead of requiring a manual stage move. Also several Driver App UI fixes so
   the Police check-in screen no longer shows Hospital-specific wording/fields.

## Found, not fixed — flagged for a decision, not silently patched

- **Stray duplicate Trip creation.** A legacy rule ("On Create - Create Trips for Regular
  Trip," Deals module) creates an orphaned Trip 5 minutes after any Police (and likely
  Hospital) per-deceased Deal is created — it predates the multi-deceased architecture and has
  no awareness these Deals already belong to a real Trip. Confirmed live during testing.
  Decided not to fix silently since it's outside today's Amber-integration scope and affects
  live automation others rely on. See `ZP-TBD-1` metadata for the full root-cause trace and a
  proposed fix direction (extend the exclusion pattern already used for the Funeral-pipeline
  sequence of the same rule).
- **`Number_of_distance_in_km` does not come back from Amber on Police cases.** Confirmed the
  Amber completion response has no distance field at all — needs a decision on whether real
  driven distance is expected from a different Amber endpoint, or manual entry.

## Security finding, not introduced today but surfaced today

Maps API keys hardcoded directly in the Deluge source rather than stored via a Zoho Connection.
See `shared/connection-notes/zoho_connections.md`. Recommend raising with Andrea directly and
migrating both to Connections, rotating the keys once moved.
