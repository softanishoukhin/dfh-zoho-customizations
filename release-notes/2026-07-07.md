# Release Notes — 2026-07-07

Backfills work completed 2026-07-03 through 2026-07-07 that was not yet captured in this repo
(last content commit was 2026-07-03, before this work happened).

## ZP-TBD-6 — Police Billing KM-Based Quantity
Fixed Police pickup and autopsy fee quantities to read from the actual trip's KM instead of a
stale Deal-level mirror / hardcoded 1. Includes a rewrite of the post-completion KM-sync path
(`updateSalesOrderOnNumberOfDistanceinKMUpdate`) and two new reusable patch functions. See
`crm/functions/deals/ZP-TBD-6_police_km_quantity_billing/`.

## ZP-TBD-7 — Police Duplicate Trip Guard
Fixed a broken (always-false) guard in `createTripFromDealForPoliceCases` that let a redundant,
empty placeholder trip get created alongside the real driver-completed trip on multi-deceased
cases. See `crm/functions/deals/ZP-TBD-7_police_duplicate_trip_guard/`.

## ZP-TBD-8 — Police Dropoffs Workflow
New feature: a "Police Dropoff" case type on the Intake app for when police physically bring 1-10
deceased directly to DFH, with no pickup trip involved. Full trip-less architecture — Account +
Deal + Operations record creation, hand tag printing, an edit-after-in-care rename cascade, and
billing that skips the pickup fee while keeping storage/autopsy/decomp fees. Built to use **zero**
new Deals workflow rules and **zero** new Deals fields, per client-stated resource limits — see
`crm/functions/deals/ZP-TBD-8_police_dropoff_workflow/metadata.md` for the full architecture
writeup and the 4 bugs found and fixed during live testing (phantom trip, Stage-mandatory
requirement, Date of Death format, Sex field omission). Widget/Creator side:
`creator/functions/nok_intake/ZP-TBD-8_police_dropoff_workflow/`.
